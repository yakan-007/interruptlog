# Team Feature Brief

作成日: 2026-06-12
対象: `app/`

## 現時点の理解

チーム対応は削除しない。ただし、まだ「何をもってチーム対応と呼ぶか」が固まりきっていない。

方針として、チーム運用は将来的にやりたい。ただし、先に個人運用を安定させる。チーム運用の具体化は、個人利用の記録・振り返り・編集・バックアップ導線が安定してから、別途じっくり打ち合わせて決める。

今のコードにあるチーム要素は、リアルタイム共同作業というより、以下の方向に近い:

- チーム共通のカテゴリ・割り込みカテゴリ・入力チップを配布する
- 配布用タスクを作り、タスクパックとしてメンバーに渡す
- 日次/月次のアーカイブ行を取り込み、メンバーやカテゴリ別に集計する
- 集中時間、割り込み、休憩の傾向をチーム単位で見る
- presence 設定の芽はあるが、現時点では本格的な同期・在席共有とは言い切れない

## チーム対応の候補

### 1. チーム標準化

チーム全員が同じ分類で記録できるようにする。

- 共通カテゴリ
- 共通割り込みカテゴリ
- 共通入力チップ
- 配布タスク

これは現在の実装に最も近い。

### 2. チーム集計

個人の記録を持ち寄って、チームの傾向を見る。

- メンバー別の集中時間
- 割り込みの多いカテゴリ
- 時間帯ごとの集中しやすさ
- 月次・年次のアーカイブ比較

これも現在の `teamArchive` / `TeamReport` に近い。

### 3. チーム運用支援

記録結果から、仕事の回し方を改善する。

- 割り込みの発生源を見える化する
- 集中時間帯をチームで守る
- 問い合わせや依頼の受け方を調整する
- タスク配布後の進捗や詰まりを把握する

ここはまだプロダクト定義が必要。

### 4. リアルタイム共有

今やっていることや状態をチームに共有する。

- 在席・集中・休憩の共有
- 現在作業中タスクの共有
- 割り込み可否の表示

これは実装コストとプライバシー設計が大きいので、今すぐ正式化しない。

## 当面の方針

- チーム機能は削除しない。
- ただし、v1 リリースからは完全に切り離し、個人運用アプリとして完成させる。
- 通常画面にはチーム導線やチーム文言を出さない。
- チーム UI は `VITE_ENABLE_TEAM_UI=true` を明示した開発時だけ表示できる扱いにする。
- 個人利用の主要導線に影響しないよう、機能フラグ配下に隔離する。
- データスキーマの `teamWorkspace` / `teamArchive` は互換性のため保持する。
- README では正式機能として強く打ち出さず、探索中であることを docs に記録する。
- リファクタリングでは、チーム機能を分割しやすい境界に寄せる。
- 当面のリファクタリングでは、チーム画面の大規模な機能追加や UX 拡張はしない。

## 個人運用が安定したとみなす目安

- タスク追加、メモ、開始、停止、完了、再開が迷わず使える。
- 割り込み、休憩、押し忘れ追加、履歴編集でメモと時刻が失われない。
- レポートと CSV / JSON バックアップで、個人ログを見返せる。
- `npm run check` と主要 App スモークテストが常に緑。
- この状態を数回の実利用で確認してから、チーム対応の打ち合わせに入る。

## 未決事項

- チーム対応の最初の価値は「標準化」「集計」「運用支援」「リアルタイム共有」のどれか
- チームデータはローカル JSON 交換で十分か、同期基盤が必要か
- メンバー名や作業内容をどこまで共有してよいか
- 個人利用モードとチーム利用モードを UI 上どの程度分けるか
- チーム機能を README に載せるタイミング
